# AGENTS.md

Инструкции для агентов, которые работают с этим репозиторием.

## Общий контекст

Репозиторий используется как учебная база по Go, backend, system design, DevOps, observability, testing, databases и interview preparation.

Основной раздел с конспектами:

```text
docs/senior-golang-study
```

Материалы должны быть практичными:
- понятно объяснять тему простыми словами;
- давать trade-offs, а не только определения;
- содержать примеры, схемы или короткие interview-ready ответы, если это помогает;
- быть полезными для подготовки к senior Go/backend собеседованиям.

## Язык и стиль

- Основной язык документации: русский.
- Английские термины можно оставлять на английском, если так их обычно используют в индустрии: `load balancer`, `gateway`, `trace`, `outbox`, `connection pool`.
- Объяснять сложные термины так, чтобы было понятно не только infrastructure/database specialist, но и backend-разработчику.
- Не писать сухой справочник без практического смысла.
- Для interview-тем добавлять короткий блок вида `Interview-ready answer` или аналогичный summary.

## Структура материалов

Если тема большая, создавать подпапку с `README.md` и отдельными файлами по подтемам.

Предпочтительный формат:

```text
some-topic/
  README.md
  01-first-concept.md
  02-second-concept.md
  03-practical-examples.md
```

Правила:
- файлы внутри подпапок нумеровать, если у них есть порядок чтения;
- `README.md` должен объяснять, что внутри, как читать и зачем это нужно;
- новые подпапки обязательно подключать в README родительского раздела;
- не создавать один огромный markdown-файл, если тема естественно делится на части;
- не плодить слишком мелкие файлы, если тема помещается в один логичный документ.

## Где размещать темы

Ориентиры:
- Go language/runtime: `01-go-core`
- архитектурные trade-offs: `04-architecture-and-patterns`
- system design схемы и request flows: `05-system-design`
- базы данных и storage: `06-databases`
- брокеры и стриминг: `07-message-brokers-and-streaming`
- HTTP/API/networking: `08-networking-and-api`
- concurrency/performance/profiling: `09-concurrency-and-performance`
- testing: `10-testing-and-quality`
- Docker/Kubernetes/observability/logs/metrics/traces: `11-devops-and-observability`
- security/secrets: `12-security`
- interview drills and answer templates: `13-interview-practice`

Если тема пересекается с несколькими разделами:
- класть основной материал туда, где он будет чаще искаться;
- из соседних разделов давать ссылку, если нужно;
- не дублировать длинные куски текста.

## Содержание документов

Для обзорных файлов по технологии использовать структуру:
- что это;
- где используется;
- сильные стороны;
- слабые стороны;
- когда выбирать;
- когда не выбирать;
- типичные ошибки;
- interview-ready answer.

Для comparison-файлов:
- добавлять таблицу;
- явно указывать trade-offs;
- не писать "X лучше Y" без контекста;
- связывать выбор с access patterns, нагрузкой, командой и operational complexity.

Для examples:
- держать примеры рядом с темой, к которой они относятся;
- не выносить все примеры в один общий файл, если удобнее читать по конкретной технологии;
- кодовые блоки должны быть короткими и объясняющими, а не production-complete.

## Оглавление в Markdown

Для длинных тематических `.md` файлов добавлять блок:

```md
## Содержание
```

Размещать его сразу после первого вводного абзаца или короткого описания файла.

Когда добавлять:
- если файл длиннее примерно 80-120 строк;
- если в файле больше 5-6 смысловых секций;
- если файл используется как справочник.

Когда можно не добавлять:
- короткая заметка;
- README, где уже есть блок `Материалы`;
- файл с одним линейным объяснением.

Формат:

```md
## Содержание

- [Где используется](#где-используется)
- [Сильные стороны](#сильные-стороны)
- [Слабые стороны](#слабые-стороны)
- [Query examples](#query-examples)
- [Interview-ready answer](#interview-ready-answer)
```

Правила:
- использовать относительные anchor-ссылки на заголовки внутри файла;
- не делать оглавление глубже одного уровня без необходимости;
- если заголовок переименован, обновить ссылку в содержании;
- не добавлять оглавление ради формальности, если оно ухудшает читаемость.

Для диаграмм:
- использовать Mermaid, если схема помогает быстрее понять flow;
- подписывать роли явно: `Cloudflare / Edge`, `Nginx reverse proxy`, `API Gateway`, `DB`, `Queue`;
- избегать абстрактных коробок без объяснения роли.

## Работа с источниками

- Для актуальных версий инструментов, библиотек и платформ сверяться с официальной документацией.
- В README подпакетов полезно оставлять ссылки на official docs.
- Не копировать большие фрагменты чужой документации; писать своими словами.

## Git

- Новые файлы сразу добавлять в git index через `git add`.
- После создания или изменения документации staging должен включать все связанные новые файлы и обновленные README.
- Не делать commit без явной просьбы пользователя.
- Не менять историю git без явной просьбы пользователя.
- Не использовать destructive git-команды вроде `git reset --hard` без явного разрешения.

## Формат ссылок

В markdown-документах использовать относительные ссылки:

```md
[Readable Title](./some-file.md)
```

В финальных ответах пользователю использовать абсолютные пути, чтобы файлы были кликабельны в среде Codex.

## Проверки

Для markdown-only изменений тесты обычно не нужны.

Но нужно проверить:
- что новые файлы находятся в правильной папке;
- что README родительского раздела обновлен;
- что нет ссылок на удаленные или переименованные файлы;
- что новые файлы staged.
