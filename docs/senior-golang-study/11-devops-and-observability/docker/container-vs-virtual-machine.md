# Container vs Virtual Machine

Это один из самых частых инфраструктурных вопросов на backend интервью.

## Что такое контейнер

Контейнер это изолированный runtime для процесса.

Проще говоря:
- приложение запускается как обычный процесс;
- но для него ограничивают namespace, filesystem, network и ресурсы;
- при этом контейнер использует ядро хоста.

То есть контейнер:
- не несет свое отдельное ядро ОС;
- не равен отдельной полноценной машине.

## Что такое виртуальная машина

Виртуальная машина это уже полноценная виртуализированная машина:
- со своей виртуальной hardware abstraction;
- со своей ОС;
- со своим kernel;
- с более тяжелым lifecycle.

## Главная разница

Контейнер:
- изоляция на уровне OS и runtime;
- общее ядро с хостом;
- легковесный;
- быстро стартует.

Виртуальная машина:
- изоляция на уровне virtual hardware;
- своя ОС и свой kernel;
- тяжелее;
- стартует дольше.

## Наглядно

VM:

```text
App -> Guest OS -> Hypervisor -> Host
```

Container:

```text
App -> Container runtime -> Host OS kernel
```

## Когда контейнеры удобны

- упаковка сервиса и зависимостей;
- reproducible runtime;
- быстрый deployment;
- orchestration через Kubernetes;
- плотное размещение сервисов на одной машине.

## Когда VM все еще уместны

- сильная изоляция;
- отдельные операционные системы;
- legacy workloads;
- инфраструктурные ограничения или compliance.

## Что важно для interview ответа

Хороший короткий ответ:
- контейнер это изолированный процесс, который делит kernel с хостом;
- VM это отдельная виртуальная машина со своей ОС;
- контейнеры легче и быстрее, VM тяжелее, но сильнее изолированы.
