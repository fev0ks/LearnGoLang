# Node Failure, Rollout And Config Delivery

## Что такое Node

`Node` это машина, на которой реально запускаются Pod'ы.

Это может быть:
- VM;
- физический сервер;
- cloud instance.

На ноде обычно есть:
- `kubelet`;
- container runtime;
- сетевой слой вроде `kube-proxy` или его замены в конкретном CNI-стеке.

## Что происходит, если нода падает

Хороший senior-level ответ:
- нода становится `NotReady`;
- Pod'ы на ней перестают быть рабочими;
- контроллеры вроде `Deployment` пересоздают Pod'ы на других нодах;
- `Service` начинает слать трафик в новые healthy Pod'ы.

Если:
- реплик несколько;
- probes настроены нормально;
- rollout strategy адекватная;

то пользователи могут почти ничего не заметить.

## Почему несколько реплик важны

Если реплика одна:
- падение Pod или Node = риск реального outage.

Если реплик несколько:
- появляется шанс на zero-downtime behavior при частичных отказах;
- `Service` может продолжить слать трафик в живые Pod'ы.

## Rollout

Для backend-сервиса важно понимать:
- как выкатится новый image;
- сколько Pod'ов можно потерять во время deploy;
- кто считается ready для приема трафика.

На практике критичны:
- readiness probe;
- liveness probe;
- rolling update strategy;
- нормальный shutdown приложения.

Если readiness настроен плохо:
- трафик может прийти в Pod слишком рано.

Если graceful shutdown нет:
- при rollout будут обрываться запросы и фоновые задачи.

## Конфиги и CI/CD

Нормальный flow обычно такой:

1. CI собирает image без environment-specific config.
2. CD деплоит манифесты.
3. Конфиги живут в `ConfigMap`, секреты в `Secret`.
4. При изменении конфигурации делается rollout, чтобы Pod'ы подхватили новые значения.

Практическое правило:
- image immutable;
- runtime config externalized;
- конфиги не должны требовать пересборки образа.

## Что полезно уметь сказать на интервью

- что произойдет при падении ноды;
- почему одной реплики обычно мало;
- как `Service` и `Deployment` помогают пережить отказ;
- как CI/CD должен обращаться с image, config и rollout;
- почему probes и graceful shutdown важны для корректного deploy.
