## Docker vs Kubernetes: что решает каждый

| | Docker | Kubernetes |
|---|---|---|
| Упаковка приложения | да | нет (использует контейнеры) |
| Запуск одного контейнера | да | да |
| Автоматический рестарт при падении | `--restart=always` | да, через Deployment |
| Несколько реплик | вручную | да |
| Service discovery | нет / вручную | встроено через Service |
| Rolling update и rollback | нет / вручную | да |
| Health checks на уровне оркестрации | нет | да (probes) |
| Управление конфигами и секретами | env files / вручную | ConfigMap, Secret |
| Управление ресурсами (CPU, memory) | `--cpus`, `--memory` | requests/limits по Pod |
| Self-healing при падении ноды | нет | да |

## Где Docker-only начинает ломаться

С одним сервисом Docker вполне работает. Проблемы появляются с ростом:

- контейнер упал — его надо поднимать вручную или настраивать через systemd/supervisord;
- нужны две реплики за балансировщиком — надо городить что-то поверх;
- rolling update без простоя — нет инструмента из коробки;
- конфиги и секреты нужно прокидывать вручную на каждый хост;
- отказ машины → все контейнеры на ней мертвы без автоматического переноса;
- разные сервисы нужно находить друг друга без hardcoded IP.

Именно этот класс проблем берет на себя Kubernetes.

## Что Kubernetes автоматизирует

**Self-healing**: если Pod упал или нода стала `NotReady`, контроллер пересоздает Pod'ы на другой ноде.

**Scaling**: указываешь желаемое число реплик — Kubernetes следит, чтобы оно всегда выдерживалось. HPA может автоматически менять число реплик по метрикам.

**Service discovery**: у каждого Service есть DNS-имя внутри кластера. Другие сервисы обращаются по имени, а не по IP.

**Rolling update**: при новом deploy Kubernetes постепенно заменяет старые Pod'ы новыми, следя за readiness probe перед тем как переключить трафик.

**Config и secrets management**: конфиг и секреты живут отдельно от image и подключаются к Pod как env vars или volume mount.

**Resource management**: CPU и memory лимиты по Pod; планировщик размещает Pod'ы по нодам с достаточными ресурсами.

## Когда Kubernetes нужен, а когда избыточен

Kubernetes — это overhead: нужно поддерживать кластер, разбираться с networking, RBAC, storage. Он оправдан, когда:

- сервисов много (десятки и больше);
- нужны zero-downtime deploy'и и rollback;
- важна автоматическая замена упавших инстансов;
- команда выросла и нужны унифицированные deploy-процессы;
- инфраструктура предоставляется как managed service (GKE, EKS, AKS).

Может быть избыточен для:
- одного-двух сервисов в маленькой команде;
- прототипов и MVP;
- задач, где достаточно managed PaaS (Cloud Run, Fly.io, Railway).

## Что должен понимать senior backend-разработчик

Cluster admin знать необязательно, но нужно уверенно понимать:

- как сервис попадает в кластер и что происходит при deploy;
- как обновление не сломает пользователей (rolling update, probes);
- как сервис переживет падение Pod или Node;
- как правильно настроить liveness и readiness probe;
- как работает graceful shutdown и почему он важен при rollout;
- как пробрасывать config и секреты в Pod;
- как requests/limits влияют на runtime Go-сервиса.

## Interview-ready answer

Docker решает упаковку и запуск одного контейнера. Kubernetes решает orchestration: следит за желаемым состоянием — нужным числом реплик, корректным rolling update, service discovery, управлением конфигами и секретами. При падении Pod или ноды Deployment controller пересоздает Pod'ы на живых нодах. Service дает стабильный DNS для балансировки трафика между репликами. Для backend-разработчика принципиально понимать probes, graceful shutdown и работу с ConfigMap/Secret — без этого deploy в Kubernetes будет ломать запросы при каждом обновлении.
