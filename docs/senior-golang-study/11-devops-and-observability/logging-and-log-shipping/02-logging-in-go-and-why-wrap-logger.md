# Logging In Go And Why Wrap Logger

Это вопрос не только про библиотеку логирования, а про то, как команда делает логи полезными в production.

## Как обычно работают с логами в Go

В Go часто используют:
- `log/slog`
- `zap`
- `zerolog`
- иногда стандартный `log`, но он быстро становится слишком примитивным

Практический production logging обычно значит:
- structured logs;
- единый набор полей;
- понятные уровни;
- correlation с trace и request id;
- единый формат во всех сервисах.

## Зачем часто делают свою обертку над логгером

Не потому что "надо завернуть все в интерфейс ради интерфейса", а потому что команде нужно единое поведение.

Своя обертка помогает:
- задать единый формат логов;
- автоматически добавлять service name, env, trace id, request id;
- скрыть детали конкретной библиотеки;
- сделать удобные helper methods для common fields;
- не размазывать logging conventions по всему коду.

## Что обычно добавляют в обертку

- `Info`, `Warn`, `Error`, `Debug`
- `With(...)`
- `WithContext(ctx)`
- добавление `trace_id`, `request_id`, `user_id`, `operation`
- конвертацию ошибок в structured fields

## Почему не хочется дергать логгер напрямую везде

Если каждый пишет лог как хочет:
- поля называются по-разному;
- в одном месте `requestId`, в другом `request_id`;
- где-то лог строкой, где-то structured;
- correlation по сервисам ломается.

## Но где опасность

Плохая обертка тоже может навредить:
- слишком толстый abstraction layer;
- потеря возможностей библиотеки;
- бессмысленный interface ради mock в тесте;
- логгер начинает жить как "магический глобальный объект".

Практическое правило:
- обертка должна упрощать conventions;
- а не прятать половину стандартного API без пользы.

## Хороший interview ответ

Своя обертка над логом полезна, когда она стандартизует поля, формат и correlation context. Она вредна, если превращается в ненужный abstraction layer без реальной пользы для команды.
