# Dynamic Programming

## Что такое DP

Задача подходит для DP, если она обладает двумя свойствами:

**Optimal substructure** — оптимальное решение задачи составляется из оптимальных решений подзадач. Пример: кратчайший путь из A в C через B — это сумма кратчайшего пути A→B и B→C.

**Overlapping subproblems** — одни и те же подзадачи встречаются многократно при рекурсивном разборе. Если подзадачи не перекрываются — это divide and conquer, а не DP.

Классическое распознавание на интервью: задача спрашивает "оптимальный способ", "минимальное/максимальное количество", "количество способов", и при наивной рекурсии дерево вызовов содержит дубликаты.

### Top-down (memoization) vs bottom-up (tabulation)

**Top-down** — рекурсивная функция с кешом результатов. Код пишется так, как думается (рекурсивно), и добавляется map или массив для запоминания уже посчитанных состояний. Подходит, когда нужна только часть состояний или когда граф зависимостей нерегулярный.

**Bottom-up** — итеративное заполнение таблицы dp[] от базовых случаев к финальному ответу. Нет overhead стека вызовов. Обычно проще оптимизировать по памяти — если рекуррентность использует только последние k строк, можно хранить только их.

Правило выбора: если сложно определить порядок вычислений (например, задача на графе с произвольными зависимостями) — начни с top-down. Если порядок очевиден (слева направо, снизу вверх) — bottom-up и с возможной оптимизацией памяти.

## Стандартный процесс мышления

Четыре шага при любой DP задаче на интервью:

1. **Определить состояние `dp[i]`** — что именно хранит эта ячейка. Это самый важный шаг. Неправильно выбранное состояние делает рекуррентность неразрешимой.

2. **Найти рекуррентное соотношение** — выразить `dp[i]` через предыдущие состояния. Обычно это одна строка кода.

3. **Определить base cases** — граничные значения, не выводимые из рекуррентности (`dp[0]`, `dp[1]` и т.д.).

4. **Определить порядок вычисления** — убедиться, что при вычислении `dp[i]` все зависимые значения уже известны.

## 1D DP задачи

### Climbing stairs

**Условие:** лестница с `n` ступеньками. За один шаг можно подняться на 1 или 2 ступени. Сколько различных способов добраться до вершины?

**Состояние:** `dp[i]` — количество способов добраться до `i`-й ступени.
**Рекуррентность:** `dp[i] = dp[i-1] + dp[i-2]`
**Base cases:** `dp[1] = 1`, `dp[2] = 2`

Шаг 1 — наивная рекурсия. Экспоненциальное время O(2^n), дерево вызовов дублируется:

```go
func climbStairsNaive(n int) int {
    if n <= 2 {
        return n
    }
    return climbStairsNaive(n-1) + climbStairsNaive(n-2)
}
```

Шаг 2 — memoization. Время O(n), память O(n):

```go
func climbStairsMemo(n int) int {
    memo := make(map[int]int)
    var dp func(i int) int
    dp = func(i int) int {
        if i <= 2 {
            return i
        }
        if v, ok := memo[i]; ok {
            return v
        }
        memo[i] = dp(i-1) + dp(i-2)
        return memo[i]
    }
    return dp(n)
}
```

Шаг 3 — tabulation. Время O(n), память O(n):

```go
func climbStairsTab(n int) int {
    if n <= 2 {
        return n
    }
    dp := make([]int, n+1)
    dp[1] = 1
    dp[2] = 2
    for i := 3; i <= n; i++ {
        dp[i] = dp[i-1] + dp[i-2]
    }
    return dp[n]
}
```

Шаг 4 — оптимизация памяти до O(1). Рекуррентность зависит только от двух предыдущих значений:

```go
func climbStairs(n int) int {
    if n <= 2 {
        return n
    }
    prev2, prev1 := 1, 2
    for i := 3; i <= n; i++ {
        curr := prev1 + prev2
        prev2 = prev1
        prev1 = curr
    }
    return prev1
}
```

### House robber

**Условие:** ряд домов, в каждом сумма `nums[i]`. Нельзя грабить два соседних дома. Найти максимальную сумму.

**Состояние:** `dp[i]` — максимальная сумма при рассмотрении первых `i` домов.
**Рекуррентность:** `dp[i] = max(dp[i-1], dp[i-2] + nums[i])`
**Base cases:** `dp[0] = nums[0]`, `dp[1] = max(nums[0], nums[1])`

Обоснование рекуррентности: для дома `i` либо не берем его (берем лучшее от предыдущих `dp[i-1]`), либо берем его (тогда предыдущий пропускаем: `dp[i-2] + nums[i]`).

```go
func rob(nums []int) int {
    n := len(nums)
    if n == 0 {
        return 0
    }
    if n == 1 {
        return nums[0]
    }

    prev2 := nums[0]
    prev1 := max(nums[0], nums[1])

    for i := 2; i < n; i++ {
        curr := max(prev1, prev2+nums[i])
        prev2 = prev1
        prev1 = curr
    }
    return prev1
}

func max(a, b int) int {
    if a > b {
        return a
    }
    return b
}
```

Время O(n), память O(1).

### Coin change

**Условие:** набор номиналов монет `coins` и сумма `amount`. Найти минимальное количество монет для набора суммы. Если невозможно — вернуть -1.

**Состояние:** `dp[i]` — минимальное количество монет для суммы `i`.
**Рекуррентность:** `dp[i] = min(dp[i-coin] + 1)` для каждой монеты `coin <= i`
**Base case:** `dp[0] = 0`

**Почему жадность не работает:** жадный алгоритм всегда выбирает самую большую монету. Для `coins = [1, 3, 4]` и `amount = 6` жадность даст `4 + 1 + 1 = 3` монеты, но оптимально `3 + 3 = 2` монеты. Жадность корректна только для канонических систем (например, обычные монеты 1, 5, 10, 25 центов), а не для произвольных наборов.

```go
func coinChange(coins []int, amount int) int {
    const inf = amount + 1
    dp := make([]int, amount+1)
    for i := 1; i <= amount; i++ {
        dp[i] = inf
    }
    dp[0] = 0

    for i := 1; i <= amount; i++ {
        for _, coin := range coins {
            if coin <= i && dp[i-coin]+1 < dp[i] {
                dp[i] = dp[i-coin] + 1
            }
        }
    }

    if dp[amount] == inf {
        return -1
    }
    return dp[amount]
}
```

Время O(amount * len(coins)), память O(amount).

Важная деталь инициализации: `dp[i] = amount + 1` (а не `math.MaxInt`) чтобы избежать переполнения при `dp[i-coin] + 1`.

### Longest increasing subsequence

**Условие:** найти длину наидлиннейшей возрастающей подпоследовательности (LIS) в массиве `nums`.

**Состояние:** `dp[i]` — длина LIS, заканчивающейся на элементе `nums[i]`.
**Рекуррентность:** `dp[i] = max(dp[j] + 1)` для всех `j < i` где `nums[j] < nums[i]`
**Base case:** `dp[i] = 1` для каждого `i` (каждый элемент сам по себе — подпоследовательность длины 1)

Решение O(n²):

```go
func lengthOfLIS(nums []int) int {
    n := len(nums)
    if n == 0 {
        return 0
    }

    dp := make([]int, n)
    for i := range dp {
        dp[i] = 1
    }

    best := 1
    for i := 1; i < n; i++ {
        for j := 0; j < i; j++ {
            if nums[j] < nums[i] && dp[j]+1 > dp[i] {
                dp[i] = dp[j] + 1
            }
        }
        if dp[i] > best {
            best = dp[i]
        }
    }
    return best
}
```

Время O(n²), память O(n).

Patience sorting позволяет решить LIS за O(n log n): поддерживается отсортированный массив "хвостов" стопок, и на каждом шаге выполняется бинарный поиск. Это полезно знать как факт, но на интервью O(n²) решение с ясной логикой часто предпочтительнее.

## 2D DP задачи

### Unique paths

**Условие:** сетка `m x n`. Робот стартует из верхнего левого угла и может двигаться только вправо или вниз. Сколько уникальных путей до правого нижнего угла?

**Состояние:** `dp[i][j]` — количество путей до клетки `(i, j)`.
**Рекуррентность:** `dp[i][j] = dp[i-1][j] + dp[i][j-1]`
**Base cases:** первая строка и первый столбец — по единице (только один способ туда попасть).

```go
func uniquePaths(m, n int) int {
    dp := make([][]int, m)
    for i := range dp {
        dp[i] = make([]int, n)
        dp[i][0] = 1
    }
    for j := 0; j < n; j++ {
        dp[0][j] = 1
    }

    for i := 1; i < m; i++ {
        for j := 1; j < n; j++ {
            dp[i][j] = dp[i-1][j] + dp[i][j-1]
        }
    }
    return dp[m-1][n-1]
}
```

**Оптимизация до 1D:** при вычислении строки `i` нужна только строка `i-1`. Текущее значение `dp[j]` в массиве уже хранит значение сверху (`dp[i-1][j]`), а `dp[j-1]` — значение слева после обновления.

```go
func uniquePathsOptimized(m, n int) int {
    dp := make([]int, n)
    for j := range dp {
        dp[j] = 1
    }

    for i := 1; i < m; i++ {
        for j := 1; j < n; j++ {
            dp[j] += dp[j-1]
        }
    }
    return dp[n-1]
}
```

Время O(m*n), память O(n).

### Longest common subsequence

**Условие:** даны две строки `text1` и `text2`. Найти длину их наидлиннейшей общей подпоследовательности (LCS). Подпоследовательность — элементы в том же порядке, но не обязательно подряд.

**Состояние:** `dp[i][j]` — длина LCS для `text1[:i]` и `text2[:j]`.
**Рекуррентность:**
- если `text1[i-1] == text2[j-1]`: `dp[i][j] = dp[i-1][j-1] + 1`
- иначе: `dp[i][j] = max(dp[i-1][j], dp[i][j-1])`

```go
func longestCommonSubsequence(text1, text2 string) int {
    m, n := len(text1), len(text2)
    dp := make([][]int, m+1)
    for i := range dp {
        dp[i] = make([]int, n+1)
    }

    for i := 1; i <= m; i++ {
        for j := 1; j <= n; j++ {
            if text1[i-1] == text2[j-1] {
                dp[i][j] = dp[i-1][j-1] + 1
            } else if dp[i-1][j] > dp[i][j-1] {
                dp[i][j] = dp[i-1][j]
            } else {
                dp[i][j] = dp[i][j-1]
            }
        }
    }
    return dp[m][n]
}
```

**Восстановление самой строки LCS** — обратный обход таблицы:

```go
func reconstructLCS(text1, text2 string, dp [][]int) string {
    i, j := len(text1), len(text2)
    result := []byte{}

    for i > 0 && j > 0 {
        if text1[i-1] == text2[j-1] {
            result = append(result, text1[i-1])
            i--
            j--
        } else if dp[i-1][j] > dp[i][j-1] {
            i--
        } else {
            j--
        }
    }

    // reverse
    for l, r := 0, len(result)-1; l < r; l, r = l+1, r-1 {
        result[l], result[r] = result[r], result[l]
    }
    return string(result)
}
```

Время O(m*n), память O(m*n). Можно оптимизировать до O(min(m,n)) если нужна только длина, а не сама строка.

### 0/1 Knapsack

**Условие:** `n` предметов с весами `weights[i]` и ценностями `values[i]`. Рюкзак вместимостью `capacity`. Каждый предмет можно взять не более одного раза. Максимизировать суммарную ценность.

**Состояние:** `dp[i][w]` — максимальная ценность при рассмотрении первых `i` предметов и вместимости `w`.
**Рекуррентность:**
- не берем предмет `i`: `dp[i][w] = dp[i-1][w]`
- берем предмет `i` (если `weights[i-1] <= w`): `dp[i][w] = dp[i-1][w-weights[i-1]] + values[i-1]`
- `dp[i][w] = max` из двух вариантов

**Почему нужно измерение по предметам:** без него мы не отследим, взяли ли уже данный предмет. В задаче "0/1" (каждый предмет не более одного раза) нельзя обойтись без этой информации. В unbounded knapsack (предметы можно брать неограниченно) измерение по предметам уже не нужно.

```go
func knapsack(weights, values []int, capacity int) int {
    n := len(weights)
    dp := make([][]int, n+1)
    for i := range dp {
        dp[i] = make([]int, capacity+1)
    }

    for i := 1; i <= n; i++ {
        for w := 0; w <= capacity; w++ {
            dp[i][w] = dp[i-1][w] // не берем предмет i
            if weights[i-1] <= w {
                take := dp[i-1][w-weights[i-1]] + values[i-1]
                if take > dp[i][w] {
                    dp[i][w] = take
                }
            }
        }
    }
    return dp[n][capacity]
}
```

**Оптимизация до 1D** — итерация по весу в обратном порядке гарантирует, что каждый предмет учитывается не более одного раза:

```go
func knapsack1D(weights, values []int, capacity int) int {
    dp := make([]int, capacity+1)

    for i := range weights {
        // обратный порядок: предотвращает повторное использование предмета
        for w := capacity; w >= weights[i]; w-- {
            take := dp[w-weights[i]] + values[i]
            if take > dp[w] {
                dp[w] = take
            }
        }
    }
    return dp[capacity]
}
```

Время O(n * capacity), память O(capacity).

Важная деталь: прямой порядок цикла по `w` дал бы unbounded knapsack, потому что обновленное `dp[w-weights[i]]` уже включает предмет `i`. Обратный порядок гарантирует, что берется значение "до" добавления предмета `i`.

## Паттерн: DP на строках

### Palindromic substrings count

**Условие:** дана строка `s`. Найти количество подстрок-палиндромов (включая однобуквенные).

Задачу можно решить через 2D DP: `dp[i][j] = true` если `s[i..j]` — палиндром. Но это O(n²) по времени и O(n²) по памяти.

Лучше — expand around center. Каждый палиндром имеет центр. Для строки длины `n` существует `2n - 1` возможных центров (каждый символ + каждый промежуток между символами). Расширяем из каждого центра, пока символы совпадают.

```go
func countSubstrings(s string) int {
    count := 0
    n := len(s)

    expand := func(left, right int) {
        for left >= 0 && right < n && s[left] == s[right] {
            count++
            left--
            right++
        }
    }

    for i := 0; i < n; i++ {
        expand(i, i)   // нечетная длина
        expand(i, i+1) // четная длина
    }
    return count
}
```

Время O(n²), память O(1) — в отличие от DP варианта с O(n²) памяти. Это пример, где DP по формальным критериям применима, но более прямое наблюдение о структуре задачи дает лучшее решение.

## Когда НЕ нужен DP

**Greedy достаточно**, если локально оптимальный выбор всегда ведет к глобально оптимальному решению:
- Activity selection (выбор максимального числа несовместимых мероприятий) — сортировка по времени окончания и жадный выбор.
- Coin change с канонической системой монет (1, 5, 10, 25) — жадность корректна, доказано математически для таких систем.
- Fractional knapsack — предметы делимы, поэтому greedy по ценности/весу оптимален.

Признак того, что greedy достаточен: можно доказать "exchange argument" — замена любого не-жадного выбора на жадный не ухудшает решение.

**Backtracking**, когда нужны не оптимальное значение, а все решения:
- Задачи типа "найти все комбинации", "перечислить все пути" — DP здесь просто не отвечает на вопрос.
- Constraint satisfaction без оптимизации.

DP и backtracking часто путают. Ключевой вопрос: нужно ли *подсчитать* или *оптимизировать* (→ DP), или нужно *перечислить* конкретные решения (→ backtracking)?

| Climbing stairs | `dp[i]` — кол-во способов добраться до ступени `i` | `dp[i] = dp[i-1] + dp[i-2]` | O(n) time, O(1) space |
| House robber | `dp[i]` — макс. сумма по первым `i` домам | `dp[i] = max(dp[i-1], dp[i-2] + nums[i])` | O(n) time, O(1) space |
| Coin change | `dp[i]` — мин. монет для суммы `i` | `dp[i] = min(dp[i-coin] + 1)` по всем `coin` | O(n * k) time, O(n) space |
| LIS | `dp[i]` — длина LIS, заканчивающейся на `nums[i]` | `dp[i] = max(dp[j] + 1)` для `j < i, nums[j] < nums[i]` | O(n²) time, O(n) space |
| Unique paths | `dp[i][j]` — кол-во путей до клетки `(i,j)` | `dp[i][j] = dp[i-1][j] + dp[i][j-1]` | O(m*n) time, O(n) space |
| LCS | `dp[i][j]` — длина LCS для `text1[:i]` и `text2[:j]` | совпадение: `dp[i-1][j-1]+1`, иначе `max(dp[i-1][j], dp[i][j-1])` | O(m*n) time, O(m*n) space |
| 0/1 Knapsack | `dp[i][w]` — макс. ценность из `i` предметов при вместимости `w` | `max(dp[i-1][w], dp[i-1][w-wt]+val)` | O(n*W) time, O(W) space |
| Palindromic substrings | центр палиндрома | expand around center | O(n²) time, O(1) space |
