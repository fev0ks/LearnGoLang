## Сравнительная таблица

| Критерий | Монолит | Modular monolith | Микросервисы |
|---|---|---|---|
| **Deployment** | Один artifact | Один artifact | N независимых сервисов |
| **Локальная разработка** | Просто | Просто | Сложно (docker-compose, service mesh) |
| **Транзакции** | ACID, легко | ACID, легко | Distributed transactions, saga |
| **Latency** | Нет network hops | Нет network hops | Каждый вызов — сеть |
| **Масштабирование** | Всё сразу | Всё сразу | Независимо по сервисам |
| **Изоляция отказов** | Нет (один процесс) | Нет | Да (при circuit breaker) |
| **Team ownership** | Сложно | По модулям | По сервисам |
| **CI/CD** | Одна pipeline | Одна pipeline | N pipeline'ов |
| **Observability** | Один лог, один трейс | Один лог, один трейс | Distributed tracing обязателен |
| **Data consistency** | Сильная (одна БД) | Сильная (одна БД) | Eventual (разные БД) |
| **Contract versioning** | Не нужна | Между модулями | API versioning, consumer-driven contracts |
| **Operational cost** | Низкий | Низкий | Высокий |

## Монолит

```
┌──────────────────────────────────────┐
│             Monolith                 │
│                                      │
│  [HTTP]─►[Service]─►[Repository]─►   │
│                                      │
│              ▼                       │
│  ┌───────────────────────────────┐   │
│  │          Database             │   │
│  └───────────────────────────────┘   │
└──────────────────────────────────────┘
```

**Когда монолит — правильный выбор:**
- Ранняя стадия продукта, домен ещё не устоялся
- Команда до 5-8 человек
- Скорость выпуска фич важнее изоляции
- Нет разных профилей нагрузки на части системы

**Плюсы:**
- Проще локальная разработка и дебаг
- Нет network latency между компонентами
- Легко делать транзакции
- Меньше DevOps overhead

**Минусы:**
- Со временем растёт coupling: всё зависит от всего
- Deployment blast radius: один баг роняет всё
- Сложнее изолировать команды при росте
- Тесты замедляются вместе с кодовой базой

## Modular monolith

```
┌────────────────────────────────────────────────────────┐
│                    Monolith Process                     │
│                                                        │
│  ┌────────────┐    ┌────────────┐    ┌────────────┐    │
│  │  Orders    │    │  Payments  │    │   Users    │    │
│  │  Module    │───►│   Module   │    │   Module   │    │
│  │            │    │            │    │            │    │
│  │ public API │    │ public API │    │ public API │    │
│  └────────────┘    └────────────┘    └────────────┘    │
│        ↑                ↑                  ↑           │
│  cross-module calls only through public interfaces     │
│                                                        │
│  ┌──────────────────────────────────────────────────┐  │
│  │              Shared Infrastructure                │  │
│  │     (logger, metrics, DB pool, config)            │  │
│  └──────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────┘
```

**Когда modular monolith — правильный выбор:**
- Продукт вырос из простого монолита
- Доменные границы уже частично понятны
- Микросервисная сложность пока не окупается
- Хочется сохранить простой deployment

**Правила модульных границ в Go:**
```go
// orders/module.go — только публичный API модуля
package orders

type Module struct {
    svc *OrderService  // internal
}

func New(db *pgxpool.Pool, events EventPublisher) *Module { ... }

// Только эти методы доступны другим модулям:
func (m *Module) CreateOrder(ctx context.Context, cmd CreateOrderCmd) (OrderID, error)
func (m *Module) GetOrder(ctx context.Context, id OrderID) (Order, error)
func (m *Module) CancelOrder(ctx context.Context, id OrderID) error
```

**Признак здорового модуля:** его можно удалить, заменить или выделить в отдельный сервис с понятным списком зависимостей.

## Микросервисы

```
                        ┌─────────────────┐
              ┌────────►│  Order Service  │────────►
              │          └─────────────────┘         │
API Gateway───┤          ┌─────────────────┐         ▼
              ├────────►│ Payment Service │────► Kafka
              │          └─────────────────┘         │
              └────────►│  User Service   │◄──────────┘
                         └─────────────────┘

Каждый сервис: своя БД, свой deployment, своя команда
```

**Когда микросервисы оправданы:**
- Команды уже достаточно автономны (Conway's Law)
- Части системы требуют независимого масштабирования
- Доменные границы стабильны и понятны
- Есть зрелость: CI/CD, observability, incident response

**Плюсы:**
- Независимый deployment и масштабирование
- Изоляция отказов (с circuit breaker)
- Team ownership по bounded contexts
- Разные технологии для разных задач

**Минусы:**
- Network latency на каждый вызов
- Distributed transactions → saga/outbox
- Distributed tracing обязателен
- Намного дороже в эксплуатации
- Локальная разработка требует инфраструктуры

## Эволюционный путь

```
Время / Сложность →

Stage 1                Stage 2                Stage 3
────────               ────────               ────────
Simple                 Modular                Microservices
Monolith          →    Monolith          →    (выборочно)

  [everything]         [orders]               [orders-svc]
                        [payments]             [payments-svc]
                        [users]                [monolith] ← остаток

Триггер для перехода:
  → Команда растёт      → Доменные границы      → Конкретная причина:
  → Домен устаялся        понятны                 scaling, team boundary,
                        → Монолит стало           разные SLA
                          трудно тестировать
```

**Типичная ошибка: выделять сервисы слишком рано**

```
Плохо: 2 разработчика → сразу 10 микросервисов
  → Большая часть времени на DevOps, не на фичи
  → При смене границ — дорого переписывать
  → Distributed transactions там где была одна

Хорошо: монолит → modular monolith → выделить только там где есть причина
  → Постепенная эволюция
  → Границы проверены на практике
  → Microservices как reward, не как стартовая точка
```

## Decision guide

```
Новый продукт?
  └── Да → Монолит. Начни просто.
  └── Нет (растущая система) →
        Есть доменные границы?
          └── Нет → Modular monolith, выдели границы сначала
          └── Да →
                Есть независимые scaling requirements?
                  └── Нет → Modular monolith достаточно
                  └── Да →
                        Команды автономны?
                          └── Нет → Modular monolith + prepare ownership
                          └── Да → Microservices для конкретных сервисов
```

**Индикаторы для микросервисов:**

| Индикатор | Пример |
|---|---|
| Разные профили нагрузки | Payment processing vs reporting |
| Разные SLA | Critical checkout vs analytics |
| Разные команды с ownership | Order team vs logistics team |
| Разные частоты деплоя | User service (ежедневно) vs billing (раз в неделю) |
| Разные scaling требования | Video encoding vs user auth |

## Когда микросервисы — ошибка

**Распределённый монолит** — худшее из двух миров:

```
service-A  ────synchronous HTTP────►  service-B
              ◄────────────────────
                  (при каждом запросе)

+ Deployment coupling: A и B нужно деплоить вместе
+ Shared database: общая схема, нет data ownership
+ No circuit breaker: падение B = падение A

Это не микросервисы. Это монолит с лишними сетевыми вызовами.
```

**Признаки распределённого монолита:**
- Сервисы нельзя деплоить независимо
- У сервисов общая база данных
- Цепочка синхронных вызовов через 5+ сервисов
- Любое изменение требует координации с другими командами

## Interview-ready answer

> "Я воспринимаю этот выбор как эволюционный, а не разовый. На ранней стадии монолит дешевле: нет distributed transactions, проще дебаг, меньше DevOps. Когда домен устаялся и команда выросла — modular monolith даёт контроль границ без операционной сложности микросервисов. Микросервисы я бы вводил точечно: только там, где есть конкретная причина — независимый scaling, разные команды с ownership, разные SLA.
>
> Главная ошибка — распределённый монолит: взять все минусы микросервисов плюс coupling монолита. Это происходит когда сервисы деплоятся вместе, используют общую БД и синхронно зависят друг от друга.
>
> Вопрос который я задаю: 'Можно ли выделить этот модуль в отдельный сервис без переписывания половины системы?' Если нет — сначала навести порядок в монолите."
