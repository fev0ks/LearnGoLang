# Browser Input And Navigation Start

Когда пользователь вводит `google.com`, цепочка начинается не с backend и даже не с DNS, а с браузера.

## 1. Браузер решает: это URL или поисковый запрос

Современный browser использует omnibox:
- если строка похожа на URL или hostname, браузер пытается открыть сайт;
- если строка похожа на обычный текст, он отправляет search query в поисковик по умолчанию.

Примеры:
- `google.com` обычно распознается как hostname;
- `golang scheduler explained` обычно распознается как search query.

## 2. Браузер разбирает URL

Если это navigation к сайту, браузер определяет:
- схему, например `https`;
- host, например `google.com`;
- порт, если он не указан явно, берется default;
- path, query string и fragment, если они есть.

Для `google.com` браузер обычно превращает navigation в:

```text
https://google.com/
```

Причина:
- сегодня почти весь публичный web ожидает `HTTPS`;
- дальше возможен redirect на canonical host, например `https://www.google.com/`.

## 3. Проверяются локальные cache и состояние браузера

До выхода в сеть browser смотрит:
- есть ли service worker;
- есть ли HSTS policy;
- есть ли cached redirect;
- есть ли закэшированные DNS hints или открытые соединения;
- нет ли готового ответа из HTTP cache.

Примеры:

HSTS:
- если браузер уже знает, что домен должен открываться только по `HTTPS`, он даже не пробует `HTTP`.

HTTP cache:
- если ресурс кэшируемый и еще свежий, браузер может не ходить в сеть за телом ответа;
- но для главной HTML-страницы это зависит от cache headers и browser policy.

## 4. Браузер создает navigation request

На этом этапе browser формирует внутреннюю задачу:
- открыть document request;
- выбрать network stack;
- подготовить security policy и origin context;
- начать resolution host name.

Важно:
- это еще не значит, что сетевой пакет уже ушел;
- сначала браузер должен получить IP и подготовить соединение.

## Что часто упускают

- ввод в адресную строку и ввод в search bar это уже отдельная логика распознавания;
- request может быть удовлетворен не только из сети, но и частично из cache;
- browser до DNS уже принимает решения, которые меняют дальнейший маршрут.

## Что могут спросить на интервью

- чем отличается URL от search query на уровне browser behavior;
- что делает HSTS;
- в каких случаях браузер не делает полноценный network round trip;
- зачем нужен browser cache и почему он не всегда спасает latency.
