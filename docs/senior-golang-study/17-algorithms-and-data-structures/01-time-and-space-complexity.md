# Временная и пространственная сложность

Сложность алгоритма — это формальный способ описать, как растёт стоимость работы
(время или память) при увеличении размера входных данных `n`.
Обозначение `O(f(n))` называется **нотацией "O большое"** (Big-O) и описывает
верхнюю границу роста в худшем случае, игнорируя константные коэффициенты и
слагаемые низшего порядка.

---

## Таблица роста: сколько операций при разных n

```
n=         1       10       100      1 000     10 000    алгоритм
---------------------------------------------------------------------------
           1        1         1          1          1    O(1)
           0        3         6          9         13    O(log n)  (log₂)
           1       10       100      1 000     10 000    O(n)
           0       33       664      9 965    132 877    O(n log n)
           1      100    10 000  1 000 000  100 000 000  O(n²)
           1    1 024      2^100      ...        ...     O(2^n)
           1  3 628 800     ...       ...        ...     O(n!)
```

Строка `O(2^n)` показывает, почему экспоненциальные алгоритмы неприменимы
даже при `n=100`: число операций больше атомов в наблюдаемой вселенной.

## Форма кривых роста

```
cost
  |                                          O(2^n)
  |                                        /
  |                                      /
  |                              O(n²) /
  |                             /    /
  |                            /   /
  |                O(n log n) /  /
  |               /          / /
  |          O(n)/          //
  |            /           /
  |           /    -------/ O(sqrt n)
  |          / ---/
  |      ---/ O(log n)
  |  ---/
  | /  O(1) ___________________________________
  |/
  +--------------------------------------------> n
```

Ключевое наблюдение: между `O(n)` и `O(n²)` огромная пропасть.
Алгоритм на 10 000 элементах делает 10 000 операций против 100 000 000.

## Все классы сложности

| Обозначение  | Название           | Пример алгоритма                         | Макс. n за 1 сек (10⁸ ops) |
|--------------|--------------------|------------------------------------------|---------------------------|
| O(1)         | Константная        | Доступ к элементу массива, хэш-таблица   | любой                     |
| O(log n)     | Логарифмическая    | Бинарный поиск, операции в B-tree        | ~10^(10^8)                |
| O(√n)        | Квадратный корень  | Факторизация числа перебором             | ~10^16                    |
| O(n)         | Линейная           | Линейный поиск, обход списка             | ~10^8                     |
| O(n log n)   | Линейно-лог.       | Merge Sort, Heap Sort, Quick Sort (avg)  | ~5 × 10^6                 |
| O(n²)        | Квадратичная       | Bubble Sort, Insertion Sort, перебор пар | ~10^4                     |
| O(n³)        | Кубическая         | Умножение матриц (наивное), Floyd-Warsh. | ~500                      |
| O(2^n)       | Экспоненциальная   | Перебор подмножеств, брутфорс TSP        | ~25-30                    |
| O(n!)        | Факториальная      | Перебор перестановок                     | ~12-13                    |

Колонка "макс. n за 1 сек" — приближение для машины с 10⁸ простых операций в секунду.
В реальных задачах это число меньше из-за кэш-промахов, выделения памяти и ветвлений.

## Лучший, средний и худший случай

Для одного алгоритма может существовать несколько разных оценок в зависимости
от характера входных данных.

- **Лучший случай (best case)** — Omega-нотация Ω(f(n)): нижняя граница.
  Пример: линейный поиск находит элемент на первой позиции → Ω(1).
- **Средний случай (average case)** — Theta-нотация Θ(f(n)): рост при типичных входных данных.
  Требует знания распределения входных данных.
- **Худший случай (worst case)** — O(f(n)): верхняя граница.
  Чаще всего именно о нём спрашивают на собеседованиях.

### Почему анализируют худший случай

Система должна работать корректно при любых входных данных, в том числе
при враждебных. Если злоумышленник или случайный пользователь может подать
входные данные, которые вызовут деградацию, именно этот сценарий критичен.

### Quick Sort — классический пример расхождения

| Случай  | Сложность  | Когда происходит                                    |
|---------|------------|-----------------------------------------------------|
| Лучший  | O(n log n) | Пивот всегда делит массив пополам                   |
| Средний | O(n log n) | Случайные входные данные                            |
| Худший  | O(n²)      | Массив уже отсортирован, пивот выбирается крайний  |

На практике средний случай Quick Sort быстрее Merge Sort из-за локальности данных,
поэтому `sort.Slice` в Go использует pdqsort (паттерн-разрушающий быстрый sort),
который специально избегает худшего случая.

## Амортизированная сложность

Амортизация — это оценка **средней стоимости** операции в серии из N операций,
даже если отдельные операции дорогие.

Формальное определение: если N операций суммарно стоят O(N × f(n)), то
амортизированная стоимость одной операции — O(f(n)).

### Пример: append в Go

Внутри Go slice хранит указатель на массив, длину и ёмкость.
Когда `len == cap`, `append` вызывает `runtime.growslice`, который:
1. Выделяет новый массив в два раза больше (примерно).
2. Копирует все элементы — это O(n).
3. Добавляет новый элемент — O(1).

Однако удвоение происходит редко. Посчитаем суммарную стоимость N append'ов:

```
Операции копирования при росте от 1 до N:
  1 + 2 + 4 + 8 + ... + N = 2N - 1 = O(N)
```

Значит, N операций append в сумме стоят O(N), то есть **амортизированно O(1)** за операцию.

```go
package main

import "fmt"

// demonstrateAmortizedAppend показывает, что реаллокация происходит O(log n) раз.
func demonstrateAmortizedAppend(n int) {
    s := make([]int, 0) // cap=0, len=0
    reallocations := 0
    prevCap := cap(s)

    for i := 0; i < n; i++ {
        s = append(s, i)
        if cap(s) != prevCap {
            reallocations++
            prevCap = cap(s)
        }
    }

    fmt.Printf("n=%d, reallocations=%d, final cap=%d\n", n, reallocations, cap(s))
}

// Вывод при n=1_000_000:
// n=1000000, reallocations=19, final cap=1048576
// 19 = ~log₂(1_000_000), то есть реаллокаций логарифмически мало.
```

Практический вывод: если известен приблизительный размер, выгодно сразу выделить
память через `make([]T, 0, expectedSize)` — это устраняет реаллокации полностью.

### Ещё один пример: таблица хэшей

Map в Go тоже амортизированно O(1) для вставки. При превышении коэффициента
загрузки происходит дорогой rehash O(n), но это случается редко.

## Пространственная сложность

### Вспомогательная vs. полная память

- **Вспомогательная (auxiliary) space** — память сверх входных данных.
  In-place алгоритм (например, Quicksort) имеет вспомогательную O(log n) (стек рекурсии).
- **Полная (total) space** — вспомогательная + память под входные данные.

На практике важна вспомогательная: входные данные уже существуют в памяти,
вопрос в том, сколько дополнительного расходует алгоритм.

### Стек рекурсии

Каждый рекурсивный вызов добавляет кадр в стек. Для дерева высоты `h`:

```go
// Обход дерева в глубину — O(h) пространства на стеке.
// В несбалансированном дереве h = O(n), в сбалансированном h = O(log n).
func dfs(node *TreeNode) {
    if node == nil {
        return
    }
    dfs(node.Left)  // рекурсивный вызов: +1 кадр на стеке
    dfs(node.Right) // рекурсивный вызов: +1 кадр на стеке
    // Стек одновременно содержит не более h кадров.
}
```

### Go-специфика

| Структура       | Размер заголовка / накладные расходы                           |
|-----------------|----------------------------------------------------------------|
| slice header    | 24 байта: указатель (8) + len (8) + cap (8)                    |
| string header   | 16 байт: указатель (8) + len (8)                               |
| map             | ~8 байт указатель + hmap struct ~200 байт + бакеты             |
| goroutine stack | ~2 KB начальный размер, растёт по мере необходимости           |
| interface       | 16 байт: тип (8) + данные (8)                                  |

Следствие: передача большого slice в функцию — это O(1) по памяти, потому что
копируется только 24-байтовый заголовок. Передача map — тоже O(1): копируется указатель.

## Сложность стандартных операций в Go

| Операция                    | Код                        | Время     | Память |
|-----------------------------|----------------------------|-----------|--------|
| Доступ по индексу           | `s[i]`                     | O(1)      | O(1)   |
| Append (амортизированно)    | `append(s, x)`             | O(1)*     | O(1)*  |
| Lookup в map                | `m[k]`                     | O(1)**    | O(1)   |
| Вставка в map               | `m[k] = v`                 | O(1)**    | O(1)   |
| Длина slice/map/string      | `len(s)`                   | O(1)      | O(1)   |
| Сортировка slice            | `sort.Slice(s, less)`      | O(n log n)| O(log n)|
| Поиск подстроки             | `strings.Contains(s, sub)` | O(n·m)*** | O(1)   |
| Send в буферизованный канал | `ch <- x`                  | O(1)      | O(1)   |

`*` — амортизированно; одиночный append при росте cap стоит O(n).

`**` — в среднем; при коллизиях деградирует, но runtime использует рандомизацию хэша
для защиты от намеренных коллизий.

`***` — n = len(s), m = len(sub); в реализации Go используется алгоритм
Рабина-Карпа / Boyer-Moore-Horspool, но в worst-case всё равно O(n·m).

## Как считать сложность составного алгоритма

### Правило 1: последовательные блоки → O(max)

```go
func example(s []int) {
    // Блок A: O(n)
    sum := 0
    for _, v := range s {
        sum += v
    }

    // Блок B: O(n log n)
    sort.Ints(s)

    // Итого: O(n) + O(n log n) = O(n log n)
    // Доминирует наибольший член.
    _ = sum
}
```

### Правило 2: вложенные циклы → O(произведение)

```go
func pairSum(s []int, target int) bool {
    for i := 0; i < len(s); i++ {         // O(n)
        for j := i + 1; j < len(s); j++ { // O(n)
            if s[i]+s[j] == target {
                return true
            }
        }
    }
    return false
    // Итого: O(n) × O(n) = O(n²)
}
```

### Правило 3: рекурсия → мастер-теорема

Для рекуррентного соотношения вида `T(n) = a·T(n/b) + f(n)`:

```
a  — количество рекурсивных вызовов
b  — на сколько частей делится задача
f(n) — стоимость работы на текущем уровне

Если f(n) = O(n^c):
  c < log_b(a)  →  T(n) = O(n^log_b(a))    // рекурсия доминирует
  c = log_b(a)  →  T(n) = O(n^c · log n)   // поровну
  c > log_b(a)  →  T(n) = O(f(n))          // внешняя работа доминирует

Merge Sort: a=2, b=2, f(n)=O(n) → c=1, log₂(2)=1 → c=log_b(a) → O(n log n)
```

### Разбор сложного примера

```go
// findDuplicatesAboveThreshold — найти дубликаты, которые встречаются > threshold раз.
// Вход: nums []int, threshold int
// Выход: []int — элементы, встречающиеся чаще threshold раз.
func findDuplicatesAboveThreshold(nums []int, threshold int) []int {
    // Шаг 1: подсчёт частот через map — O(n) time, O(n) space
    freq := make(map[int]int, len(nums))
    for _, v := range nums { // O(n) итераций
        freq[v]++            // O(1) амортизированно
    }

    // Шаг 2: сбор результатов — O(n) итераций по map
    result := make([]int, 0)
    for val, count := range freq {
        if count > threshold { // O(1)
            result = append(result, val) // O(1) амортизированно
        }
    }

    // Шаг 3: сортировка для детерминированного порядка — O(k log k),
    // где k — количество уникальных элементов выше порога, k <= n.
    sort.Ints(result) // O(k log k) = O(n log n) в worst case

    return result
}

// Итоговая сложность:
//   Time:  O(n) + O(n) + O(n log n) = O(n log n)
//   Space: O(n) для freq map + O(k) для result = O(n)
```

Обратите внимание: наличие `map` внутри цикла не ухудшает сложность с O(n)
до O(n²), потому что операции с map амортизированно O(1).

## Практические ориентиры для backend-разработчика

### Когда сложность реально бьёт

**O(n²) на 10k элементах = 100 000 000 операций.**
Это уже секунды задержки в обработчике HTTP-запроса.
Типичный сценарий: вложенный цикл для поиска совпадений в двух списках.
Решение: map для одного из списков → O(n).

**N+1 query problem = O(n) запросов к БД вместо O(1).**
Загрузить список из 1000 пользователей, затем для каждого сделать запрос
профиля — это 1001 запрос. Даже при 1ms на запрос — секунда задержки.
Решение: JOIN или batch-загрузка → O(1) запрос.

**JSON/Protobuf marshaling = O(n) по размеру payload.**
Нельзя игнорировать: 10MB JSON-ответ будет сериализоваться O(10MB).
При высоком RPS это существенная нагрузка на CPU.

**Сортировка для пагинации = O(n log n) при каждом запросе.**
Если сортировать 100k записей при каждом запросе страницы — это дорого.
Решение: индексированный столбец в БД, сортировка на уровне хранилища.

**Конкатенация строк в цикле.**
В Go строки иммутабельны, каждая `s += piece` создаёт новую строку.
За N итераций суммарно копируется O(n²) байт.
Решение: `strings.Builder` → O(n) суммарно.

```go
// Плохо: O(n²) суммарно
func joinBad(parts []string) string {
    result := ""
    for _, p := range parts {
        result += p // каждый раз копирует всё накопленное
    }
    return result
}

// Хорошо: O(n) суммарно
func joinGood(parts []string) string {
    var b strings.Builder
    for _, p := range parts {
        b.WriteString(p) // записывает в внутренний буфер без лишних копий
    }
    return b.String()
}
```

## Хороший ответ на интервью

Когда интервьюер просит оценить сложность алгоритма, сильный ответ включает
следующие компоненты:

- **Назвать сложность явно** — не "этот алгоритм быстрый", а "time O(n log n),
  space O(n) вспомогательная".

- **Разделить worst / average / best case** — особенно если они различаются,
  как у Quick Sort. Объяснить, при каких входных данных наступает худший случай.

- **Обосновать** — пройтись по коду и указать, какой блок вносит какой вклад:
  "внешний цикл O(n), внутренний O(log n) из-за бинарного поиска, итого O(n log n)".

- **Упомянуть trade-off по памяти** — часто временную сложность можно улучшить
  за счёт памяти (хэш-таблица вместо повторного поиска) и наоборот.

- **Связать с реальным backend-контекстом** — показать понимание, при каком `n`
  алгоритм начнёт проседать на продакшн-нагрузке, и предложить альтернативу.
