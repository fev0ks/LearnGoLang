# Garbage Collector

GC в Go нужен, чтобы освободить разработчика от ручного управления памятью, но для senior-уровня важно понимать цену этого удобства.

Главный practical takeaway:
- проблема обычно не в самом наличии GC;
- проблема в высоком allocation rate, лишних временных объектах и неконтролируемом росте live heap.

## Ментальная модель

Упрощенно GC в Go:
- отслеживает достижимые объекты;
- периодически помечает живые объекты;
- освобождает недостижимые;
- старается делать это конкурентно с работой приложения.

Для production reasoning полезны три понятия:
- allocation rate;
- live heap;
- GC overhead.

Если сервис постоянно создает много мусора:
- GC запускается чаще;
- CPU уходит на marking/scanning;
- latency становится менее предсказуемой.

## Что важно знать senior-разработчику

### GC не чинит memory leaks автоматически

Если на объект все еще есть ссылка:
- для GC он живой;
- даже если по бизнес-смыслу он уже "не нужен".

Классические источники удержания памяти:
- long-lived maps и caches без eviction;
- slice, удерживающий большой backing array ради маленького окна данных;
- goroutine leaks, удерживающие ссылки на контекст и буферы;
- глобальные структуры и очереди без лимитов.

### Важен не только объем памяти, но и churn

Сервис может потреблять умеренно памяти, но создавать огромное число короткоживущих объектов.

Это бьет по:
- CPU;
- tail latency;
- общему throughput под нагрузкой.

### Stack growth дешевле, чем heap churn

Если данные могут жить на стеке, это обычно выгоднее.
Поэтому темы `escape analysis` и API design напрямую связаны с GC-поведением.

## Что обычно помогает

- переиспользование буферов там, где это оправдано;
- снижение числа временных аллокаций на горячем пути;
- ограничение размеров cache и очередей;
- аккуратная работа со `slice`, чтобы не удерживать большой массив;
- проверка allocation profile вместо догадок.

## Что часто помогает меньше, чем кажется

- агрессивные `sync.Pool` без измерений;
- ручные микрооптимизации вне hot path;
- попытка лечить GC-тюнингом плохую архитектуру хранения данных.

`sync.Pool` полезен, когда:
- объекты действительно часто создаются;
- reuse хорошо укладывается в профиль нагрузки;
- есть измеримый выигрыш.

Но он не должен становиться "дефолтным ответом" на все memory-проблемы.

## Что смотреть в диагностике

- heap profile;
- allocation profile;
- GC pause и GC CPU fraction;
- рост RSS и live objects под нагрузкой;
- число и длительность goroutines, которые удерживают память.

Инструменты:
- `go tool pprof`;
- `runtime.ReadMemStats`;
- `/debug/pprof/heap`;
- benchmark с `-benchmem`.

## Типовые production-сценарии

### Сервис с высокой QPS и JSON

Проблема:
- много короткоживущих аллокаций на decode/encode.

Что делать:
- мерить `allocs/op`;
- сокращать временные объекты;
- не переусложнять API ради гипотетического выигрыша.

### Worker с большими батчами

Проблема:
- редкие, но крупные всплески heap usage;
- удержание памяти между батчами.

Что делать:
- смотреть, не держат ли slice и map старые capacity;
- контролировать размеры батчей;
- проверять, где остаются long-lived references.

## Что могут спросить на интервью

- почему высокий allocation rate ухудшает latency даже без OOM;
- чем memory leak в Go отличается от утечки в языках с manual memory management;
- когда стоит использовать `sync.Pool`, а когда нет;
- как найти код, который удерживает память слишком долго.
