# Escape Analysis

Escape analysis определяет, может ли значение безопасно жить на стеке или его нужно разместить в heap.

## Базовая идея

- стек дешевле для allocation и cleanup;
- heap дороже, потому что объект нужно потом отслеживать и убирать GC;
- компилятор решает размещение по факту использования значения, а не по тому, написал ты `new` или `&x`.

Важно:
- указатель сам по себе не означает heap;
- значение может уйти в heap без явного указателя;
- решение принимает компилятор, а не программист "по интуиции".

## Типовые причины escape

- возврат указателя на локальное значение;
- сохранение значения в интерфейс, если компилятору трудно доказать безопасность;
- замыкания, которые переживают текущий stack frame;
- передача значения в другую goroutine;
- слишком сложный data flow, который мешает анализу компилятора.

## Как смотреть escape analysis

Основная команда:

```bash
go build -gcflags=-m ./...
```

На что смотреть:
- `moved to heap`;
- `escapes to heap`;
- сообщения об inlining, потому что он тоже влияет на результат анализа.

Важно не читать вывод буквально без контекста:
- сам факт escape не всегда проблема;
- важна частота аллокаций на горячем пути;
- иногда более чистый API важнее одной лишней аллокации.

## Что реально дает выигрыш

- уменьшение временных объектов на hot path;
- снижение allocation churn в JSON/HTTP/database слоях;
- меньше работы для GC;
- более стабильная latency под нагрузкой.

## Что часто пытаются "оптимизировать" зря

- переписывают читаемый код ради нулевой аллокации вне hot path;
- заменяют интерфейсы concrete types без измеримого эффекта;
- боятся возвращать указатель там, где это нормальный и понятный API;
- гоняются за каждым escape вместо профилирования.

## Практический подход

1. Найди hot path через pprof или benchmark.
2. Посмотри allocation profile.
3. Сверь с `-gcflags=-m`.
4. Исправляй только то, что дает измеримый эффект.

## Что могут спросить на интервью

- почему локальная переменная может оказаться в heap;
- означает ли `new(T)` автоматическое размещение в heap;
- почему передача через интерфейс иногда ухудшает allocation profile;
- как ты проверяешь, что оптимизация реально сработала.
