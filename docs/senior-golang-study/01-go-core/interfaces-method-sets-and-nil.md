## Как interface устроен концептуально

Интерфейсное значение содержит:
- конкретный динамический тип;
- данные этого значения или указатель на них.

Из этого следует два важных вывода:
- `interface == nil` только когда и type, и data отсутствуют;
- интерфейс может быть не `nil`, даже если внутри лежит typed nil pointer.

## Классическая nil-ловушка

Проблемный случай:

```go
var p *MyError = nil
var err error = p
fmt.Println(err == nil) // false
```

Почему так:
- у `err` есть конкретный динамический тип `*MyError`;
- значит сам interface уже не `nil`.

На практике это часто ломает:
- возврат ошибок;
- optional dependencies;
- кастомные интерфейсные адаптеры;
- logging и metrics, когда nil-check ожидается "по смыслу", а не по реальному representation.

## Method sets

Нужно помнить:
- методы со value receiver входят в method set и значения, и указателя;
- методы с pointer receiver входят только в method set указателя.

Следствие:
- `T` может удовлетворять интерфейсу, а `*T` тоже может;
- но если интерфейс требует метод с pointer receiver, значение `T` уже не подойдет.

Это влияет на:
- дизайн интерфейсов;
- return types фабрик;
- работу generics с интерфейсными ограничениями;
- неожиданности при передаче concrete types в интерфейс.

## Почему маленькие интерфейсы лучше

В Go интерфейс обычно описывает поведение потребителя, а не поставщика.

Хороший признак:
- интерфейс маленький;
- живет ближе к месту использования;
- не тащит лишние методы "на будущее".

Плохой признак:
- огромный service interface ради mock;
- общий интерфейс в пакете-поставщике без реальной необходимости;
- интерфейс вводится до появления второй реализации.

## Production-антипаттерны

- возврат `error`, внутри которого typed nil;
- интерфейсы уровня "god object" с десятком методов;
- принятие интерфейса там, где concrete type проще и безопаснее;
- скрытая мутабельность через интерфейсные методы;
- зависимость на интерфейс ради тестов вместо нормальной композиции.

## Практические правила

- возвращай `nil`, а не typed nil в интерфейсе;
- объявляй интерфейсы со стороны потребителя;
- предпочитай маленькие capability-interfaces;
- если API зависит от мутации состояния, внимательно выбирай pointer/value receiver.
