# Memory Model

Memory model отвечает на вопрос: когда запись в одной goroutine гарантированно видна другой goroutine.

Это не академическая тема. Без нее невозможно уверенно рассуждать о race conditions, visibility bugs и корректности concurrent-кода.

## Главная идея

Если между двумя операциями нет корректной synchronization edge, нельзя полагаться на порядок наблюдения изменений.

Иначе говоря:
- "на моей машине работает" не значит "код корректен";
- простое чтение и запись из разных goroutines без синхронизации уже проблема;
- даже если значение выглядит примитивным, visibility все равно не гарантируется.

## Happens-Before

Ключевое понятие: happens-before.

Если операция A happens-before B, то эффекты A гарантированно видны для B.

Практические источники happens-before:
- отправка в channel и соответствующее получение;
- unlock `sync.Mutex` перед последующим lock;
- операции из `sync/atomic`, если они используются корректно;
- `WaitGroup`, `Once` и другие synchronization primitives с документированными гарантиями.

## Чего memory model не прощает

- чтение shared state без lock, "потому что запись редкая";
- double-checked logic без корректной синхронизации;
- публикацию указателя на partially initialized объект;
- надежду, что порядок операций сохранится "сам по себе".

Даже если panic нет:
- значение может быть старым;
- часть полей может наблюдаться в несогласованном состоянии;
- баг может проявляться только под нагрузкой или на другой архитектуре.

## Channels не заменяют все

Channels дают сильные guarantees, но сами по себе не являются универсальной заменой mutex.

Обычно:
- channels хороши для передачи ownership, сигналов, pipeline coordination;
- mutex и atomic лучше для защиты shared mutable state на hot path.

Плохая идея:
- использовать channel там, где нужен обычный короткий critical section;
- использовать mutex там, где проще передать владение данными одной goroutine.

## Atomic operations

`sync/atomic` полезен, когда:
- нужен очень маленький lock-free coordination primitive;
- есть простой инвариант;
- команда понимает ограничения и цену этого подхода.

Плохая идея:
- строить сложную многопольную логику на atomic без четкой модели публикации и чтения;
- использовать atomic просто потому, что "mutex медленнее".

На senior-интервью важно проговаривать:
- atomic снижает overhead только в определенных сценариях;
- reasoning about correctness часто сложнее, чем с mutex.

## Data races

Data race возникает, когда:
- два конкурентных доступа идут к одной памяти;
- хотя бы один из них запись;
- между ними нет корректной синхронизации.

Последствия:
- undefined behavior на уровне ожиданий программы;
- нестабильные, трудно воспроизводимые баги;
- ложное ощущение корректности до первой реальной нагрузки.

Инструмент первой линии:

```bash
go test -race ./...
```

Но нужно помнить:
- race detector находит только то, что реально было исполнено;
- он не заменяет reasoning о корректности.

## Практические правила

- если данные shared и mutable, синхронизация должна быть явной;
- если можно убрать shared ownership, лучше убрать его;
- document invariants около concurrent state;
- выбирай самый простой synchronization primitive, который сохраняет корректность.

## Что могут спросить на интервью

- что значит happens-before простыми словами;
- почему чтение bool без lock тоже может быть некорректным;
- когда использовать channel, mutex и atomic;
- почему race detector полезен, но недостаточен.
