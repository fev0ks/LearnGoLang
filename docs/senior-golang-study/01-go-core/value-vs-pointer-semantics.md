## Базовая идея

- В Go почти все передается по значению.
- Даже когда ты передаешь указатель, по значению передается сам указатель.
- Для `map`, `slice`, `chan`, `func`, `interface` копируется маленький header, который ссылается на underlying data.

Нужно уметь различать:
- копию самого значения;
- копию дескриптора, который все еще смотрит на общие данные;
- передачу указателя на изменяемое состояние.

## Что копируется, а что остается общим

`struct`:
- копируется целиком;
- если внутри есть поля-ссылки вроде `slice`, `map`, `pointer`, то копия struct все равно может делить часть состояния с оригиналом.

`slice`:
- копируется header: pointer, len, cap;
- underlying array обычно остается общим, пока append не вызовет realocation.

`map`:
- копируется runtime descriptor;
- обе переменные смотрят на одну и ту же map-структуру.

`interface`:
- копируется пара "type + data";
- это может выглядеть как "легкое значение", но внутри может жить указатель на общие данные.

## Когда лучше value semantics

- объект маленький и логически immutable после создания;
- хочется уменьшить shared mutable state;
- нужно упростить reasoning и тестирование;
- значение часто живет на стеке и не должно утекать в heap;
- API должен явно показывать, что функция не меняет входной аргумент.

Примеры:
- configuration snapshots;
- small domain structs;
- time values, IDs, money-like value objects.

## Когда лучше pointer semantics

- структура большая и дорогая для копирования;
- метод должен менять receiver;
- нужно различать "значение отсутствует" и "значение равно zero value";
- тип содержит synchronization primitives и его нельзя безопасно копировать;
- объект должен иметь стабильную identity.

Примеры:
- mutable service state;
- cache entries;
- large buffers;
- structs с `sync.Mutex`, `sync.RWMutex`, `sync.Once`, atomic-полями.

## Частые production-ошибки

- копирование struct с mutex и последующая работа с двумя копиями блокировки;
- неявное разделение underlying array между несколькими slice;
- возврат указателя там, где значение было бы безопаснее и проще;
- хранение указателей на временные переменные в циклах и неожиданный aliasing;
- API, где часть методов меняет состояние, а часть нет, но это не видно по receiver.

## Практические правила

- если тип логически value object, начинай с value semantics;
- если тип содержит lock или shared mutable state, почти всегда избегай копирования;
- receiver-стиль должен быть консистентным внутри типа;
- если сомневаешься, смотри не только на perf, но и на aliasing complexity.

## Что могут спросить на интервью

- почему `slice` выглядит как value type, но часто ведет себя как reference-like;
- когда pointer receiver обязателен, а когда это просто привычка;
- почему копирование struct с mutex опасно;
- в каких случаях передача по значению уменьшает вероятность race condition.
