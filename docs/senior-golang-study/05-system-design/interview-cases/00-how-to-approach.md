## Что оценивает интервьюер

Не правильный ответ — правильного нет. Оценивается:

| Что | Почему важно |
|---|---|
| **Умение задавать вопросы** | В реальности требования всегда неполные |
| **Структурированность мышления** | Большие системы строятся по частям |
| **Знание трейдоффов** | Нет универсально правильного решения |
| **Обоснование решений** | "Я взял Redis потому что..." ≠ "взял Redis" |
| **Коммуникация** | Умение объяснить техническое нетехническому |
| **Итеративность** | Начинать простым, усложнять постепенно |

Что **не** оценивается:
- Знание конкретных продуктов наизусть (Kafka vs RabbitMQ — нюансы второстепенны)
- Скорость рисования диаграмм
- Идеальный дизайн с первой попытки

## Тайминг: 45-60 минут

```
 0:00─────── 5-10 мин ───────10-15 мин ────15-25 мин ────25-45 мин ────45-50 мин
   Уточнение     Оценка       Высокоур.       Deep          Wrap-up
   требований    нагрузки     дизайн          dive          + вопросы
       [1]         [2]           [3]           [4]            [5]
```

> **Важно:** Не переходи к рисованию до конца фазы 1. Интервьюеры ценят когда кандидат не торопится.

---

## Фаза 1: Уточнение требований (5-10 мин)

### Цель

Сузить задачу до решаемой. Никогда не проектировать "YouTube вообще" — проектировать "видеохостинг для 10M пользователей с акцентом на upload и playback".

### Что спрашивать: функциональные требования

```
Кто пользователи системы?
  → "Это B2C или внутренний инструмент?"
  → "Какая аудитория: зарегистрированные или анонимные?"

Какие core use cases?
  → "Давайте я назову что считаю обязательным, вы поправите?"
  → Назвать 3-4 функции и согласовать

Что НЕ входит в scope?
  → "Нам нужно проектировать авторизацию или можем взять как данность?"
  → "Аналитика и отчёты — в scope?"

Приоритеты?
  → "Если не успеем всё — что важнее?"
```

### Что спрашивать: нефункциональные требования

```
Масштаб:
  → "Сколько пользователей сейчас / через год?"
  → "Сколько запросов в секунду ожидаем?"
  → "Read-heavy или write-heavy?"

Надёжность:
  → "Какой SLA? 99.9% / 99.99%?"
  → "Допустима ли потеря данных?"

Консистентность:
  → "Пользователь должен сразу видеть своё действие?"
  → "Eventual consistency приемлема?"

Задержка:
  → "Что критично по latency? Какой p99 приемлем?"

Geography:
  → "Один регион или глобальное развёртывание?"
```

### Выход из фазы 1

После уточнений — вслух резюмировать:

```
"Итак, я понял задачу так: нам нужна система X для Y пользователей,
которая умеет [F1, F2, F3]. Из нефункциональных: 99.9% uptime,
p99 < 200ms, read-heavy соотношение 10:1. Аналитику и монетизацию
оставляем за скобками. Правильно?"
```

## Фаза 2: Оценка нагрузки (5 мин)

### Цель

Понять порядок чисел — это определит выбор хранилища, нужность кеша, шардирования, CDN.

### Техника "back-of-envelope"

Не нужна точность. Нужен порядок: единицы, десятки, сотни.

```
DAU (daily active users) = 10M

Reads:
  10M users × 10 requests/day = 100M requests/day
  100M / 86400 sec ≈ 1200 RPS среднее
  Peak = 3x среднее ≈ 3500 RPS

Writes:
  Соотношение read:write = 10:1
  350 writes/sec

Storage (данные):
  1 record = ~1 KB
  350 writes/sec × 86400 × 365 = ~11B записей/год
  11B × 1 KB ≈ 11 TB/год

Storage (медиа, если есть):
  Отдельно: 1 видео = 500 MB → другой порядок
```

### Что делать с числами

После подсчёта — делать выводы вслух:

```
"1200 RPS в среднем — один сервер справится, но пиковые 3500 RPS
при latency < 200ms требуют горизонтального масштабирования.
11 TB/год — реляционная база справится, но через 3-5 лет нужно
думать о шардировании. Кеш будет критичен — read-heavy."
```

### Числа которые стоит помнить

```
Операции по скорости:
  L1 cache       ~0.5 ns
  RAM            ~100 ns
  SSD read       ~100 µs
  Network (LAN)  ~0.5 ms
  HDD seek       ~10 ms
  Network (WAN)  ~100 ms

Объёмы:
  1M users × 1 field (UUID) = ~16 MB
  1B records × 100 bytes = ~100 GB

Время:
  1 day = 86,400 sec ≈ 100K sec
  1 year = 31.5M sec ≈ 30M sec
```

## Фаза 3: Высокоуровневый дизайн (10-15 мин)

### Цель

Нарисовать компоненты и потоки данных на "салфетке". Пока не детали — только что с чем общается.

### Как рисовать

```
Client → API Gateway → Service → DB
                  ↓
               Cache
                  ↓
             Message Queue → Worker
```

Стандартные компоненты — можно называть обобщённо:
- "Реляционная БД" (потом уточнить PostgreSQL/MySQL)
- "Кеш" (потом уточнить Redis/Memcached)
- "Очередь сообщений" (потом Kafka/SQS/RabbitMQ)
- "Объектное хранилище" (S3/GCS)

### Что обязательно показать

```
1. Клиент → как попадает в систему (DNS, LB, API GW)
2. Основной happy path (как данные создаются и читаются)
3. Хранилище (где данные живут)
4. Асинхронные компоненты (если есть)
```

### Говорить вслух

Не молчать пока рисуешь:

```
"Клиент идёт через API Gateway — там будет auth и rate limiting.
Дальше — сервис приложения. Для хранения я пока скажу 'база данных',
чуть позже объясню почему выбрал реляционную. Для быстрых reads
добавлю кеш поверх. Если операция асинхронная — через очередь."
```

## Фаза 4: Deep Dive (15-20 мин)

### Цель

Углубиться в самые важные или самые сложные части. Обычно интервьюер сам спрашивает — или можно предложить самому.

### Типичные темы для deep dive

```
Хранилище:
  - Какую БД и почему
  - Схема данных (основные таблицы/коллекции)
  - Индексы
  - Шардирование если нужно

Кеширование:
  - Что кешировать
  - Cache invalidation strategy
  - Что делать при cache miss

Надёжность:
  - Что если упадёт компонент X
  - Репликация, failover
  - Idempotency для операций записи

Масштабирование:
  - Horizontal scaling
  - Bottlenecks
  - SPOF (single point of failure)

Консистентность:
  - Distributed transactions
  - Eventual consistency
  - Read-your-writes гарантия
```

### Как вести deep dive

Не ждать вопросов — предлагать направление:

```
"Я хочу остановиться на двух вещах которые считаю критичными для
этой системы: схема хранилища и стратегия кеширования.
Начну со схемы — хорошо?"
```

## Фаза 5: Финал (5 мин)

```
Резюмировать решение:
  "Мы спроектировали X. Ключевые решения: [1], [2], [3].
  Трейдоффы которые я принял осознанно: [...]"

Отметить что за пределами scope:
  "За рамками остались: мониторинг, billing, analytics.
  При реальной реализации начал бы с добавления метрик с первого дня."

Предложить улучшения:
  "Если масштаб вырастет в 10x — здесь нужно будет шардировать,
  а кеш заменить на distributed кеш с eviction policy."
```

## Универсальный чеклист

Вещи которые стоит упомянуть **почти в любой задаче**:

```
□ Rate limiting — защита API от перегрузки
□ Auth / AuthZ — хотя бы упомянуть ("берём как данность JWT")
□ Idempotency — для write операций (retry safe)
□ Monitoring / Alerting — метрики, что алертить
□ Graceful degradation — что если кеш недоступен
□ Data backup / retention policy
□ CDN — если есть статика или медиа
□ Health checks — для load balancer
```

## Типичные ошибки

### Ошибка 1: Начать рисовать до уточнения требований

```
Плохо: "Окей, значит нам нужна URL shortener... вот клиент, вот база..."
Хорошо: "Прежде чем начну — пара вопросов. Ожидаемый объём?"
```

### Ошибка 2: Одно решение без альтернатив

```
Плохо: "Я возьму PostgreSQL."
Хорошо: "Возьму PostgreSQL — данные структурированы, нужны транзакции.
         Если бы нагрузка была выше или схема менее предсказуема — 
         рассмотрел бы NoSQL."
```

### Ошибка 3: Игнорировать failures

```
Плохо: Нарисовать happy path и остановиться.
Хорошо: "Что если этот сервис упадёт? Тут нужна репликация и
         failover. Клиент должен получить degrade experience, 
         а не 500."
```

### Ошибка 4: Нырять в детали сразу

```
Плохо: Начать с SQL-схемы до того как понятна общая архитектура.
Хорошо: Высокоуровневый дизайн → согласование → затем детали.
```

### Ошибка 5: Молчать

```
Плохо: Тихо рисовать 5 минут.
Хорошо: Постоянно комментировать: "Сейчас думаю о том, нужен ли здесь
         кеш. Соотношение reads 10:1, так что да — добавлю."
```

## Шаблон вопросов к интервьюеру

Хорошие вопросы показывают что ты думаешь как инженер в реальной компании:

```
"Есть ли уже какие-то legacy компоненты которые нужно учитывать?"
"Какую consistency model считаете приоритетной — strong или eventual?"
"Есть ли требования к регионам данных (GDPR, data residency)?"
"Что сейчас является bottleneck в существующей системе?"
"Какой stack используется в команде — это повлияет на выбор технологий?"
```
